Much of the study of programming languages depends on trees or, more generally, inductively defined structures.
This is in part why functional programming languages like OCaml are well-suited for the implementation of programming languages.
This means we have to spend some time on the humble notion of trees.
If you've taken a course in discrete mathematics, you've likely seens the graph-theoretic definition of a tree.

\begin{definition}
  A \textbf{tree} is undirected graph which is connected an acyclic.
  A \textbf{directed tree} is a directed graph whose underlying graph is a tree.
  A directed tree is \textbf{rooted} if there is a unique vertex with in-degree 0.
\end{definition}
%
In our setting we'll be primarily interested in \textit{nonempty rooted directed} trees; moving forward \enquote{tree} we will always mean \enquote{nonempty rooted directed tree.}
These are also sometimes called \textbf{rose trees}.
To make this more explicit we'll work with the \textit{inductive} definition of a (rose) tree.

\begin{definition}
  \label{def:rose-tree}
  A \textbf{tree} with values from $V$ is defined inductively as follows:
  \begin{itemize}
  \item if $v$ is a value from $V$ and $T_1, \dots, T_k$ are trees then so is $\tnode(v, T_1, \dots, T_k)$
  \end{itemize}
  Note, in particular, that $\tnode(v)$ a tree for any value $v$ from $V$.
  We call this kind of node a \textbf{leaf}.
  The \textbf{root} of a tree is defined as:
  \begin{align*}
    \troot {\tnode(v, T_1, \dots, T_k)} = v
  \end{align*}
  and the \textbf{children} a tree are defined as:
  \begin{align*}
    \tchildren{\tnode(v, T_1, \dots, T_k)} = [T_1, \dots, T_k]
  \end{align*}
\end{definition}
%
We can, of course, define rose trees very naturally in OCaml:
\begin{ocaml}
  type 'a tree = Node of 'a * 'a tree list
  let example = Node (1, [Node (2, []); Node (3, [])])
\end{ocaml}

\begin{example}
  \label{ex:decision-tree}
  We can use decision trees to represent boolean functions, and we can use view decisions trees as a particular class of rose tree.
  The decision tree for the boolean function $\mathsf{OR}(x_1, x_2, x_3) = x_1 \lor x_2 \lor x_3$ can be written:
  \begin{align*}
    \tnode(x_1 = 1, \tnode(1), \tnode(x_2 = 1, \tnode(1), \tnode(x_3 = 1, \tnode(1), \tnode(0))))
  \end{align*}
  Values of this tree are queries to the inputs of the function.
  Roughly speaking, this decision tree expresses the idea that the value of $\mathsf{OR}(x_1, x_2, x_3)$ can be determined by searching for left to right for an input which is equal to 1.
\end{example}

We can visualize trees in the usual way.
Here, for example, is a visualization of the tree from \Cref{ex:decision-tree}:

\begin{center}
  \begin{forest}
    for tree={
      minimum width=1in,
    }
    [{$x_1 = 1$}
      [$1$]
      [{$x_2 = 1$}
        [$1$]
        [{$x_3 = 1$}
          [$1$]
          [$0$]
        ]
      ]
    ]
  \end{forest}
\end{center}

We'll also occassionally use the following form of tree visualization, which we'll call \textbf{compact form}.

\begin{center}
  \begin{forest}
    for tree={
      grow'=0,
      child anchor=west,
      parent anchor=south,
      anchor=west,
      calign=first,
      edge path={
        \noexpand\path [draw, \forestoption{edge}]
        (!u.south west) +(5pt,0) |- (.child anchor)\forestoption{edge label};
      },
      before typesetting nodes={
        if n=1
        {insert before={[,phantom]}}
        {}
      },
      before computing xy={l=20pt},
    }
    [{$x_1 = 1$}
      [$1$]
      [{$x_2 = 1$}
        [$1$]
        [{$x_3 = 1$}
          [$1$]
          [$0$]
        ]
      ]
    ]
  \end{forest}
\end{center}
%
This form is based on visualizations of file trees, e.g., see the directory listing program \texttt{tree}.
It's particularly useful when a tree it too wide to draw otherwise, e.g., when the values in nodes are themselves very wide.\footnote{
The other benefit of this form is we can implement our own version of \texttt{tree} which generates visualizations of rose trees in this form.

\begin{example}
